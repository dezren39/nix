
# Lootbox: Configurable Values Audit & Recommendations

## Config Structure

All configuration lives in `lootbox.config.json` with four root keys:

```json
{
  "server": { },
  "client": { },
  "global": { },
  "hazmat": {
    "server": { },
    "client": { },
    "global": { }
  }
}
```

- **`server`** — Settings the server reads on boot. Client cannot set these.
- **`client`** — Settings the client uses. Delivered to client via server on-connect message or data dir. Client CLI overrides these.
- **`global`** — Settings used by both server and client (e.g., port).
- **`hazmat`** — Internal constants nobody should touch. Same `server`/`client`/`global` sub-structure. Not documented in help, not in example configs. Embedded as a `defaults.json` in the compiled binary or as a constants value.

All config is read by server on boot and doesn't change at runtime. Client config is delivered by server (on-connect message sends client config) and can be overridden by client CLI flags or a local config file in the data dir.

### Defaults

A `defaults.json` (or constants object) is embedded into the binary containing all default values. This is the single source of truth for defaults — no more scattered magic numbers.

## Permissions (formerly "sandbox")

### Rename
- Config key: `permissions` (under `server`)
- Remove `sandbox` as a name everywhere
- CLI: `--no-sandbox` removes default permissions AND sets `--allow-all` (full Deno permissions)

### Deno terminology
Deno calls these **"Permissions"**. 10 `--allow-*` flags exist: `all`, `read`, `write`, `net`, `env`, `sys`, `run`, `ffi`, `import`, `scripts`. Each (except `all`/`scripts`) accepts scoped values and has a `--deny-*` counterpart.

### Config values accepted

| Value | Behavior |
|---|---|
| `true` | Use current defaults (`--allow-net`, `--allow-import=localhost:PORT`) |
| `false` / `null` | No permissions at all (no `--allow-*` flags besides `--allow-import`) |
| `"all"` | Full permissions (`--allow-all`) |
| `"net,read,env"` (string) | `.split(",")` → trimmed → each becomes `--allow-{value}` |
| `["net", "read=/tmp", "env=HOME"]` (array) | Each item parsed in order into args. `null`/`false`/empty-string filtered out. |
| `{}` (object) | See object format below |

### Object format for permissions

```json
{
  "defaults": true,
  "allow": {
    "net": true,
    "read": "/tmp,/var",
    "env": ["HOME", "PATH"],
    "write": null
  },
  "deny": {
    "ffi": true
  }
}
```

Rules:
- `defaults` (boolean): if `true`, prepend current defaults (`--allow-net`, `--allow-import`). If `false`, disable them. Applied iteratively in arrays: `[{"defaults": false}, {"defaults": true}]` → defaults enabled.
- `allow`/`deny`: if either exists, they must be the only keys besides `defaults`. If neither exists, other keys are interpreted as `allow` keys.
- Each allow/deny key value handling:
  - `true` → blanket permission with no scope: `--allow-{key}` (or `--deny-{key}`)
  - `""` (empty string) → same as `true`: blanket `--allow-{key}` (no scope)
  - `null` / `false` → **no effect**, filtered out (this key produces no flag)
  - `string` (non-empty) → single scoped value: `--allow-{key}={string}`
  - `string[]` → flattened, `null`/`false`/`""` items filtered. Each remaining string → `--allow-{key}={string}`. If all items filter out (empty list after filtering) → blanket `--allow-{key}`.
- Map key order: try to follow JSON order. Allow keys emitted before deny keys.

### Array format for permissions

```json
[
  { "defaults": false },
  { "allow": { "net": true, "read": "/tmp" } },
  { "deny": { "write": "/etc" } },
  "env",
  { "defaults": true }
]
```

Each item parsed in order into args. String items are shorthand for `--allow-{item}`. No override/merge — what you type is what you get.

### CLI flags for permissions

Accept ALL of these, parsed in arg order, appended after config values:

| CLI Flag | Effect |
|---|---|
| `--allow net,read,env` | `--allow-net --allow-read --allow-env` |
| `--allow-net` | `--allow-net` |
| `--allow-read=/tmp` | `--allow-read=/tmp` |
| `--deny-write` | `--deny-write` |
| `--deny-ffi=/usr/lib` | `--deny-ffi=/usr/lib` |
| `--no-sandbox` | Remove default permissions AND add `--allow-all`. Deno with `--allow-all` grants full access — this is equivalent to running unsandboxed. (Note: Deno without ANY permission flags will deny all access and prompt interactively, but since lootbox runs scripts non-interactively, no flags = no permissions = scripts fail on any privileged operation.) |

Config values cannot be removed by CLI — CLI args are appended after config args. Deno resolves validity of the final combination.

## Bugs/Issues to Fix

| Issue | Description | Fix |
|---|---|---|
| `--timeout`/`--no-sandbox` not forwarded | `server.ts` only forwards port/root/data-dir to `get_config`. `--timeout`, `--no-sandbox`, `--allow-*`, `--deny-*` are silently ignored when running `lootbox server --timeout X` | Add all new flags to `server.ts` parseArgs and forwarding logic |
| `RPC_WORKER_CODE` inline timeout | `worker_manager.ts` has 30s hardcoded in inlined worker string. Can't read config at runtime. | Pass `rpcTimeout` as CLI arg to worker subprocess. Default to `server.timeout` if `rpcTimeout` not set separately. |
| Default port scattered | `3000` appears in 6 files | Extract to `DEFAULT_PORT` constant in defaults/constants, reference everywhere. |
| Client timeout vs script timeout | `exec.ts` hardcodes 30s. If server timeout is 30 min, client gives up at 30s. | `clientTimeout = max(timeout + clientTimeoutBuffer, 30000)`. Buffer is configurable (any int). Server sends client config on-connect. |
| Client config delivery | No mechanism for server to tell client about config values (autoDisconnectDelay, clientTimeout, etc.) | Server sends client config in WebSocket welcome message. Client can also read config from data dir. Client CLI overrides both. |

## Example Config File

```json
{
  "global": {
    "port": 3000,
    "lootboxRoot": ".lootbox"
  },
  "server": {
    "timeout": 1800000,
    "permissions": ["net", "read", "env=HOME"],
    "rpcTimeout": 60000
  },
  "client": {
    "serverUrl": "ws://localhost:3000/ws",
    "clientTimeout": 1805000,
    "clientTimeoutBuffer": 5000,
    "autoDisconnectDelay": 100
  }
}
```

Hazmat is never shown in examples. Users who know, know.
