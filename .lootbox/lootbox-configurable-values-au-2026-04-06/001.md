
# Lootbox: Configurable Values Audit & Recommendations

## Answering Your Questions First

### 1. CLI flags: server or client?

The current `--timeout` and `--no-sandbox` flags are parsed in `get_config.ts`, which is called:
- **Server-side**: by `server.ts` → `get_config()` → used in `execute_llm_script.ts` (where timeout/sandbox actually apply)
- **Client-side**: by `exec.ts` → `get_config()` (but timeout/sandbox aren't used there — client has its own hardcoded 30s WebSocket timeout)

**Problem**: `server.ts` only forwards `--port`, `--lootbox-root`, and `--lootbox-data-dir` to `get_config()`. It does NOT forward `--timeout` or `--no-sandbox`. So currently `--timeout` only works if passed as a raw CLI arg (e.g., `deno run ... --timeout 30000`), not via `lootbox server --timeout 30000`. The config file path works fine for both.

### 2. Deno's --allow-* flags — what are they called?

Deno officially calls them **"Permissions"** (not "sandbox capabilities"). The docs page is titled "Security and permissions." The term "sandbox" is informal.

**All 10 Deno permission flags:**

| Flag | Short | Accepts scoped values? | Description |
|---|---|---|---|
| `--allow-all` | `-A` | No | All permissions (disables sandbox) |
| `--allow-read[=<PATH>...]` | `-R` | Yes (paths) | Filesystem read |
| `--allow-write[=<PATH>...]` | `-W` | Yes (paths) | Filesystem write |
| `--allow-net[=<HOST>...]` | `-N` | Yes (hosts:ports) | Network access |
| `--allow-env[=<VAR>...]` | `-E` | Yes (var names, supports wildcards) | Environment variables |
| `--allow-sys[=<API>...]` | `-S` | Yes (API names) | System info |
| `--allow-run[=<PROG>...]` | — | Yes (program names) | Subprocess spawning |
| `--allow-ffi[=<PATH>...]` | — | Yes (library paths) | FFI/NAPI |
| `--allow-import[=<HOST>...]` | — | Yes (hostnames) | Remote imports |
| `--allow-scripts` | — | No | npm post-install scripts |

There are also `--deny-*` counterparts (deny takes precedence over allow).

### 3. Sandbox config design: `true`/`false`/string/array?

**Recommendation: Yes, this makes sense.** The design should be:

| Value | Behavior |
|---|---|
| `true` (default) | Current sandbox: `--allow-net` only |
| `false` | No sandbox: `--allow-all` |
| `"net,read,env"` (string) | Specific permissions: `--allow-net --allow-read --allow-env` |
| `["net", "read=/tmp", "env=HOME"]` (array) | Specific permissions with optional scoped values |

Keep `--no-sandbox` as CLI shorthand for `false`.

For CLI granularity: **`--allow net,read,env`** (single flag, comma-separated) is simpler than `--allow-net --allow-read --allow-env` (which would conflict with Deno's own flags if someone runs the source directly). The config file array form gives full power; CLI gives convenient shortcuts.

### Summary of Bugs/Issues Found

| Issue | Description | Fix |
|---|---|---|
| `--timeout` not forwarded | `server.ts` only forwards port/root/data-dir to `get_config`. New flags like `--timeout` and `--no-sandbox` are silently ignored when running `lootbox server --timeout X` | Add `--timeout` and `--no-sandbox` to `server.ts` parseArgs and forwarding logic |
| `RPC_WORKER_CODE` inline string | `worker_manager.ts` has a huge inlined worker script as a string constant. The 30s timeout inside it cannot read from config at runtime. | Either pass timeout as a CLI arg to the worker, or accept that RPC timeout is separate from script timeout |
| Default port scattered | `3000` appears in 6 different files | Extract to a shared constant (e.g., `DEFAULT_PORT = 3000` in types.ts or constants.ts) |
| Client timeout independent of script timeout | `exec.ts` has 30s hardcoded, but if user sets `timeout: 1800000` (30 min), client will timeout at 30s while server is still running the script | `clientTimeout` should default to `max(timeout + 5000, 30000)` or be independently configurable |
