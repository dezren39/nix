# Plan: MCP Server Health Monitoring, Auto-Restart, and Multi-Client Conflict Prevention

## Problem Statement

Lootbox currently has **zero lifecycle management for MCP servers**. When an MCP server (like `chrome-devtools-mcp`) is configured:

1. **No health checks** — If an MCP server process dies, connection drops, or the backing service (Chrome) crashes, lootbox doesn't notice. Calls silently fail with "server not connected."
2. **No auto-restart** — Workers (tool processes) have exponential backoff + circuit breaker restart logic, but MCP servers have none. Connect once → fail forever.
3. **No multi-client conflict prevention** — Multiple lootbox instances can configure the same MCP server (e.g., `chrome-devtools-mcp` on port 9222), spawning competing processes that fight over the same Chrome instance.
4. **Shallow health endpoint** — `GET /health` returns `{"status":"ok"}` with no awareness of subsystem health. No CLI health command exists.

---

## Scope

Changes **inside lootbox** only. We do NOT modify `chrome-devtools-mcp` or Chrome itself.

---

## Phase 1: MCP Server Health Monitoring & Auto-Restart

**Goal**: Give MCP servers the same resilience that workers already have.

### 1A. Add `McpHealthMonitor` class

**File**: `src/lib/external-mcps/mcp_health_monitor.ts` (new)

A health monitor that periodically checks MCP server connections and triggers reconnection:

**Behavior**:
- Periodically pings each connected MCP client (using the MCP SDK's `ping()` method if available, or a lightweight `tools/list` call as a health probe)
- On failure, marks server as `disconnected` and initiates reconnection with exponential backoff
- On successful reconnect, re-fetches schemas and notifies the integration manager
- Emits events: `server:healthy`, `server:unhealthy`, `server:reconnecting`, `server:reconnected`, `server:failed`
- Circuit breaker: after max attempts, marks as `failed` and stops retrying

- Add connection state enum: `connected | disconnected | reconnecting | failed`

### 1C. Wire health monitor into `McpIntegrationManager`

### 1E. Add config support — per-MCP server AND global defaults

**Files**: `src/lib/lootbox-cli/types.ts`, `src/lib/get_config.ts`, `src/lib/external-mcps/mcp_config.ts` (modify)

Health/reconnect settings can be configured at **two levels**:

**1. Global defaults** (in `hazmat.server`) — applied to all MCP servers:
```json
{
  "hazmat": {
    "server": {
      "mcpHealthCheckInterval": 30000,
      "mcpMaxReconnectAttempts": 5,
      "mcpReconnectBackoffBase": 2000,
      "mcpMaxReconnectBackoff": 60000,
      "mcpHealthCheckTimeout": 5000
    }
  }
}
```

**2. Per-server overrides** (in each MCP server config) — override globals for that server:
```json
{
  "mcpServers": {
    "chrome-devtools": {
      "command": "npx",
      "args": ["chrome-devtools-mcp@latest", "--browserUrl", "http://127.0.0.1:9222"],
      "health": {
        "checkInterval": 15000,
        "maxReconnectAttempts": 10,
        "reconnectBackoffBase": 1000,
        "maxReconnectBackoff": 30000,
        "checkTimeout": 3000
      }
    },
    "some-other-server": {
      "command": "npx",
      "args": ["some-mcp-server"],
      "health": {
        "checkInterval": 60000,
        "maxReconnectAttempts": 3
      }
    }
  }
}
```

This way, a flaky server like chrome-devtools-mcp can have aggressive retry settings, while a stable server can use longer intervals.

The `McpServerConfig` type in `mcp_config.ts` gets an optional `health?: McpHealthConfig` field. Resolution: per-server `health` field → global `hazmat.server` values → constant defaults.

1. Connects to a running lootbox server via HTTP `GET /health`
2. Pretty-prints the deep health status with color coding (green/yellow/red)
3. Returns exit code 0 for `ok`, 1 for `degraded`, 2 for `unhealthy`
4. Supports `--json` flag for machine-readable output
5. Uses configurable server URL (defaults to `http://localhost:{port}/health`)

## Phase 3: MCP Multi-Client Session Management with Auto-Port

**Goal**: Enable multiple simultaneous MCP connections without conflicts. Auto-assign unique ports for MCP servers that support it (like chrome-devtools-mcp). Track sessions so a reconnecting client can reuse its existing MCP server.

### 3A. MCP Session Registry

**File**: `src/lib/external-mcps/mcp_session_registry.ts` (new)

A registry that tracks active MCP server sessions across lootbox instances:

**Storage location**: Uses lootbox's existing XDG-compliant data directory (`lootbox_data_dir` or platform default from `db.ts` — `~/Library/Application Support/lootbox/` on macOS, `$XDG_DATA_HOME/lootbox/` on Linux). Specifically: `{data_dir}/mcp-sessions/` directory.

**Registry file**: `{data_dir}/mcp-sessions/registry.json`

```typescript
interface McpSessionEntry {
  serverName: string;           // e.g., "chrome_devtools"
  sessionId: string;            // unique: `${serverName}-${pid}-${timestamp}`
  pid: number;                  // lootbox process PID
  port: number;                 // assigned port (e.g., 9222, 9223)
  originalPort: number;         // what was configured (e.g., 9222)
  startedAt: string;            // ISO timestamp
  lastHeartbeat: string;        // ISO timestamp, updated by health monitor
  lootboxPort: number;          // which lootbox instance (port 3000, 3001, etc.)
  workdir: string;              // which project directory
}

interface McpSessionRegistry {
  version: 1;
  sessions: McpSessionEntry[];
}
```

**Behavior**:
- On startup, read registry and clean stale entries (PID dead = remove)
- Update `lastHeartbeat` during health checks (doubles as liveness proof)
- On shutdown, remove own entries
- File-level locking via `Deno.flock` (or atomic rename pattern) for multi-process safety

### 3B. Auto-Port Assignment

**File**: `src/lib/external-mcps/mcp_port_assigner.ts` (new)

When an MCP server config references a specific port (detected by scanning args for patterns like `--browserUrl`, `--port`, `127.0.0.1:NNNN`), the auto-port logic:

1. Check registry for existing sessions on that port
2. If no conflict → use original port
3. If conflict → auto-increment port (9222 → 9223 → 9224...) until finding a free one
4. Rewrite the MCP server args with the new port before spawning
5. Register the session with assigned port

**Per-server configuration** (extends the `McpServerConfig`):

```json
{
  "mcpServers": {
    "chrome-devtools": {
      "command": "npx",
      "args": ["chrome-devtools-mcp@latest", "--browserUrl", "http://127.0.0.1:9222"],
      "multiClient": {
        "strategy": "auto-port",
        "portRange": [9222, 9299],
        "portArgPattern": "--browserUrl"
      }
    },
    "filesystem-server": {
      "command": "npx",
      "args": ["@modelcontextprotocol/server-filesystem"],
      "multiClient": {
        "strategy": "per-session"
      }
    }
  }
}
```

**Strategies**:
- `"warn"` (default): Log a warning when conflicts are detected but proceed. This is safe — user is informed.
- `"fail"`: Refuse to connect if a conflicting session exists on the same port.
- `"auto-port"`: Auto-increment port. Best for chrome-devtools-mcp and similar port-bound servers. Spawns a new Chrome on the new port automatically.
- `"per-session"`: Each lootbox session spawns its own independent MCP server process (already the default for stdio transport — this just makes it explicit and documented). No port conflict possible since each process is isolated.

**For `auto-port` with chrome-devtools-mcp specifically**: The server needs a Chrome instance on the new port. Two sub-options:
- If `--browserUrl` is specified, lootbox rewrites the URL port. The user must ensure Chrome is running on that port (or the MCP server's default behavior of launching its own Chrome kicks in).
- If no `--browserUrl`, the MCP server launches its own Chrome with a temp profile — no conflict at all.

### 3C. Session reuse on reconnect

When a lootbox client reconnects (e.g., after a timeout or transient failure), the health monitor checks the session registry:

- If a session entry exists with matching `sessionId` and the MCP process is still alive → reuse it (reconnect to existing transport)
- If the session is dead → clean it up and create a new one

The `sessionId` is a composite: `{serverName}-{lootboxPid}-{timestamp}` so it's unique per lootbox instance. A reconnecting client within the same process reuses its session ID.

## Risk Assessment

- **Low risk**: All changes are additive. Existing behavior is preserved — health monitoring defaults to on (30s interval), multi-client defaults to `warn` (just logs, doesn't change behavior).
- **MCP SDK compatibility**: We use `ping()` or `tools/list` for health probing. Both are standard MCP protocol operations.
- **Session registry location**: Uses lootbox's existing XDG-compliant data directory (macOS: `~/Library/Application Support/lootbox/mcp-sessions/`, Linux: `$XDG_DATA_HOME/lootbox/mcp-sessions/`, or `lootbox_data_dir` if configured). Same pattern as the SQLite database.
- **File locking**: Registry uses atomic write (write temp file → rename) for multi-process safety. Stale entries cleaned on startup via PID liveness check.
- **Auto-port**: Conservative port range defaults (9222-9299 = 78 slots). Only activated when explicitly configured as `"auto-port"` strategy. The default `"warn"` strategy doesn't change any ports — it just informs the user.
- **Performance**: Health checks every 30s are negligible. Registry I/O is minimal (one JSON file, updated on heartbeat).
