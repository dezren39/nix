# Plan: MCP Server Health Monitoring, Auto-Restart, and Multi-Client Conflict Prevention

## Problem Statement

Lootbox currently has **zero lifecycle management for MCP servers**. When an MCP server (like `chrome-devtools-mcp`) is configured:

1. **No health checks** — If an MCP server process dies, connection drops, or the backing service (Chrome) crashes, lootbox doesn't notice. Calls silently fail with "server not connected."
2. **No auto-restart** — Workers (tool processes) have exponential backoff + circuit breaker restart logic, but MCP servers have none. Connect once → fail forever.
3. **No multi-client conflict prevention** — Multiple lootbox instances or parallel sessions can configure the same MCP server (e.g., `chrome-devtools-mcp` on port 9222), spawning competing processes that fight over the same Chrome instance.
4. **Shallow health endpoint** — `GET /health` returns `{"status":"ok"}` with no awareness of MCP server health, worker health, or subsystem status.

This is a real problem for users running `chrome-devtools-mcp` across multiple projects/sessions, where CDP only allows one client per browser target.

---

## Scope

This plan covers changes **inside lootbox** only. We do NOT modify `chrome-devtools-mcp` or Chrome itself. We make lootbox resilient to MCP server failures and smart about preventing multi-client conflicts.

## Phase 1: MCP Server Health Monitoring & Auto-Restart

**Goal**: Give MCP servers the same resilience that workers already have.

### 1A. Add `McpHealthMonitor` class

**File**: `src/lib/external-mcps/mcp_health_monitor.ts` (new)

A health monitor that periodically checks MCP server connections and triggers reconnection:

```typescript
export interface McpHealthConfig {
  healthCheckIntervalMs: number;     // default 30_000 (30s)
  maxReconnectAttempts: number;      // default 5 (0 = unlimited)
  reconnectBackoffBaseMs: number;    // default 2_000
  maxReconnectBackoffMs: number;     // default 60_000
  healthCheckTimeoutMs: number;      // default 5_000
}
```

**Behavior**:
- Periodically pings each connected MCP client (using the MCP SDK's `ping()` method if available, or a lightweight `tools/list` call as a health probe)
- On failure, marks server as `disconnected` and initiates reconnection with exponential backoff
- On successful reconnect, re-fetches schemas and notifies the integration manager
- Emits events: `server:healthy`, `server:unhealthy`, `server:reconnecting`, `server:reconnected`, `server:failed`
- Circuit breaker: after max attempts, marks as `failed` and stops retrying

### 1C. Wire health monitor into `McpIntegrationManager`

## Phase 2: Deep Health Endpoint

**Goal**: Make `GET /health` actually useful.

### 2A. Enhance health response

**File**: `src/lib/rpc/managers/openapi_route_handler.ts` (modify)

Change from:
```json
{"status": "ok"}
```

To:
```json
{
  "status": "ok|degraded|unhealthy",
  "uptime_ms": 123456,
  "subsystems": {
    "workers": {
      "status": "ok",
      "total": 5,
      "ready": 5,
      "failed": 0,
      "crashed": 0
    },
    "mcp_servers": {
      "status": "ok|degraded|unhealthy",
      "servers": {
        "chrome_devtools": {
          "status": "connected|disconnected|reconnecting|failed",
          "last_health_check": "2026-04-07T...",
          "reconnect_attempts": 0
        }
      }
    }
  }
}
```

**Logic**:
- `ok`: All subsystems healthy
- `degraded`: Some MCP servers or workers unhealthy, but core functioning
- `unhealthy`: Critical failures

## Phase 3: MCP Port Conflict Detection & Prevention

**Goal**: Prevent multiple lootbox instances from fighting over the same MCP server.

### 3A. Add port/process conflict detector

**File**: `src/lib/external-mcps/mcp_conflict_detector.ts` (new)

Before connecting to an MCP server, check for conflicts:

1. **Port scanning**: For configs with `--browserUrl` or similar args pointing to a specific port, check if another lootbox process already has an MCP server connected to that port
2. **Lock file mechanism**: Create a lock file at `~/.lootbox/mcp-locks/<server-name>-<port>.lock` containing the PID and timestamp
3. **Stale lock cleanup**: If the PID in the lock file is dead, remove it and proceed
4. **Conflict resolution strategy** (configurable):
   - `warn` (default): Log a warning but proceed (user's choice)
   - `fail`: Refuse to start the conflicting MCP server
   - `auto-port`: For chrome-devtools-mcp specifically, auto-increment the port (9222 → 9223 → ...) — future enhancement

```typescript
export interface ConflictCheckResult {
  hasConflict: boolean;
  conflictingPid?: number;
  conflictingPort?: number;
  lockFile?: string;
  suggestion?: string;
}
```

### 3B. Integrate conflict detection into `McpClientManager`

**File**: `src/lib/external-mcps/mcp_client_manager.ts` (modify)

- Call conflict detector before `connectClient()`
- Acquire lock on successful connection
- Release lock on disconnect
- Log clear warnings when conflicts are detected

### 3C. Add conflict resolution config

**File**: `src/lib/lootbox-cli/types.ts` (modify)

Add to `HazmatServerExtras`:
```typescript
mcpConflictStrategy?: "warn" | "fail";
```

**File**: `test/mcp_conflict_test.ts` (new)

## Risk Assessment

- **Low risk**: All changes are additive. Existing behavior is preserved — health monitoring and conflict detection are opt-in via config.
- **MCP SDK compatibility**: We use `ping()` or `tools/list` for health probing. Both are standard MCP protocol operations.
- **Lock files**: Use `~/.lootbox/mcp-locks/` directory. Stale lock cleanup prevents deadlocks. Lock files are advisory only in `warn` mode.
- **Performance**: Health checks every 30s are negligible. Lock file I/O is minimal (one file per MCP server).
